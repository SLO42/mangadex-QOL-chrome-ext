# v0.2 — Native Chip Parity

**Date:** 2026-04-27
**Status:** approved (verbal), implementing
**Builds on:** [v0.1 design](./2026-04-27-mangadex-latest-enhancer-design.md)

## Goal

Make the injected pills visually indistinguishable from MangaDex's native chips on `/titles/recent`, while keeping the v0.1 information improvements (`+N` overflow tooltip, always-show rating chip including `safe`).

## Reference markup (captured from `/titles/recent`)

A `tags-row` from a real card:

```html
<div class="flex flex-wrap gap-1 tags-row tags self-start" style="max-height: calc(1em + 0rem);">
  <span class="inline-flex items-center gap-1 rounded uppercase text-[0.625rem] font-bold px-[0.375rem] leading-[1.5em] my-auto text-white bg-status-red">Erotica</span>
  <a href="/tag/.../romance" class="inline-flex items-center gap-1 rounded uppercase text-[0.625rem] font-bold px-[0.375rem] leading-[1.5em] my-auto bg-accent">Romance</a>
  ...
</div>
```

Decoded utility classes:

| Concern | Class |
|---|---|
| Layout (chip) | `inline-flex items-center gap-1` |
| Shape | `rounded` (4px, **not** full pill) |
| Typography | `uppercase font-bold text-[0.625rem] leading-[1.5em]` |
| Padding | `px-[0.375rem]` (6px H, 0 V — line-height does the work) |
| Vertical alignment | `my-auto` |
| Tag bg | `bg-accent` |
| Rating bg (Erotica) | `text-white bg-status-red` |

## Strategy

MangaDex's Tailwind classes are global on the page; injected elements inherit them automatically. **Apply MangaDex's class strings directly to my elements** rather than recreating their values in my own stylesheet. This guarantees pixel parity and survives MangaDex theme changes (light/dark, custom themes).

## Per-element changes

| Element | Classes |
|---|---|
| `.mdx-ext-pills` container | `mdx-ext-pills flex flex-wrap gap-1 self-start` |
| Chip base | `mdx-ext-pill inline-flex items-center gap-1 rounded uppercase text-[0.625rem] font-bold px-[0.375rem] leading-[1.5em] my-auto` |
| Tag chip | base + `bg-accent` |
| Rating chip | base + `text-white` + `bg-status-{green\|orange\|red}` |
| `+N` overflow | base + `bg-accent` + `mdx-ext-tag--more` (custom for opacity/cursor) |

**Why keep the `mdx-ext-*` prefixed classes?** Two reasons:
1. The `mdx-ext-pills` container needs `margin-top` / `margin-bottom` that MangaDex's native version doesn't have (their tags row is part of a card layout that already provides spacing; we're injecting into a row layout that doesn't).
2. The `+N` chip needs `cursor: help` and reduced opacity that no native chip has.

## Rating → bg token mapping

| Rating | Token | Confirmed? | Color (fallback) |
|---|---|---|---|
| `safe` | `bg-status-green` | guess | `#2e7d32` |
| `suggestive` | `bg-status-orange` | guess | `#ef6c00` |
| `erotica` | `bg-status-red` | ✓ from user-provided HTML | `#c62828` |
| `pornographic` ("MATURE") | `bg-status-red` | ✓ from user HTML — native UI labels pornographic content as "Mature" with the same red as Erotica; no visual distinction beyond the label | `#c62828` |

## Tag group → bg token mapping

Native MangaDex colors tag chips by their `group` attribute. We store `{name, group}` per tag (cache version 2) and apply the appropriate class.

| Group | Token | Confirmed? | Color (fallback) |
|---|---|---|---|
| `genre`, `theme` | `bg-accent` | ✓ from user HTML (Romance, Reincarnation) | (uses MangaDex's accent var; no fallback set — relies on bg-accent being in their bundle) |
| `format` | `text-white bg-status-purple` | ✓ from user HTML (Doujinshi) | `#7c3aed` |
| `content` | `text-white bg-status-red` | guess (warnings like Gore, Sexual Violence) | `#c62828` |

If a guessed token has been tree-shaken from MangaDex's bundle, the chip will render without background. **Mitigation**: keep my v0.1 `mdx-ext-rating--{rating}` classes and color rules as fallback. They're loaded by the extension's content_script CSS (after MangaDex's bundle), so they win source order — but only matter when the Tailwind class produces no background.

To avoid my fallback always overriding MangaDex's tokens, my fallback CSS uses `:where()` to keep specificity at zero:

```css
:where(.mdx-ext-rating--safe) { background-color: #2e7d32; }
```

Now any actual `bg-status-green` (specificity 0,1,0) wins over my zero-specificity fallback. If `bg-status-green` is missing, fallback applies.

## Files touched

- `src/content.js` — update `pillRenderer.render()` to apply the new class strings.
- `src/styles.css` — shrink to: container margins, `+N` opacity/cursor, fallback rating colors via `:where()`.

## Test plan

1. Reload extension, reload `/titles/latest`.
2. Visually compare a chip on `/titles/latest` with the same chip on `/titles/recent` — should match border-radius, padding, font, color.
3. Inspect a tag chip — confirm `bg-accent` is resolving to a non-empty `background-color` in computed styles.
4. Inspect a rating chip for safe / suggestive / erotica / pornographic — confirm each has a background. If safe/suggestive look unstyled, the token was tree-shaken and the fallback should be kicking in (verify in computed styles).
5. Hover the `+N` chip — cursor should be help-style and tooltip should list overflow tags.
6. Toggle light/dark mode in MangaDex — chip colors should follow the theme (this is the bonus from using their tokens).

## Out of scope

- Making tag chips clickable links to `/tag/{id}/{slug}` (would require knowing each tag's UUID at render time; currently we only have names from the cached payload). Punt to v0.3 if needed.
- Matching their `max-height: calc(1em + 0rem)` truncation (rejected per user choice — `+N` tooltip is more useful for `/titles/latest`).
