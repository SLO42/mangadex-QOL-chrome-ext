# v0.3 — Library-Status Stripes

**Date:** 2026-04-27
**Status:** approved (verbal), spec for review
**Builds on:** v0.1, v0.2

## Goal

On every listing page, mark each row with a colored left-edge stripe indicating the user's reading status for that manga (Reading, Plan to Read, Completed, On Hold, Re-reading, Dropped). Manga not in the user's library get no stripe.

## Non-goals

- Modifying the library (read-only).
- Distinguishing "in library without status" from "not in library" (none of these get stripes in v0.3).
- Status icon overlay on the cover (deferred to v0.3.1).
- A settings UI to customize colors (hardcoded defaults).
- Refresh-token rotation when the access token expires (user can reload the MangaDex tab to recover).

## Architecture

```
src/
├── content.js          # extends with stripeRenderer + auth discovery + scope expansion
├── background.js       # extends with libraryFetcher + librarySession state
├── styles.css          # extends with .mdx-ext-status--* color rules
```

No new files — keeping the v0.1/v0.2 module split. Auth discovery sits in `content.js` because `localStorage` is a content-script concern (the SW can't read page-origin storage).

### Component additions

| Component | File | Responsibility |
|---|---|---|
| `findJWT()` | content.js | Scans `mangadex.org` localStorage for JWT-shaped values. Decodes the payload of each candidate; returns the one whose claims look like a MangaDex auth token (`aud` / `iss` / `client_id` containing "mangadex"). Returns `null` if not logged in. |
| `tokenSync` | content.js | On boot, calls `findJWT()`. If found, sends `{type: SET_TOKEN, token}` to SW. On 401-from-SW, re-runs discovery and retries once. |
| `libraryFetcher` | background.js | Calls `GET /manga/status` with `Authorization: Bearer {token}`. Parses response, stores `{statuses, fetchedAt}` in `chrome.storage.session` under key `library:status`. |
| `libraryCache` | background.js | 5-minute TTL gate around `libraryFetcher`. Returns cached statuses if fresh. |
| `stripeRenderer` | content.js | After v0.1's row scan finds a `(anchor, mangaId)` pair, asks SW for that manga's status (batched along with the existing `LOOKUP_MANGA` request — see "Message protocol" below). On response, applies `mdx-ext-status mdx-ext-status--{status}` to the anchor. |

## Auth flow

```
content.js boot
  └─ findJWT()
       │
       ├─ found → sendMessage(SET_TOKEN)
       │            SW writes to chrome.storage.session (auto-cleared on browser close)
       │            SW calls libraryFetcher → caches statuses
       │
       └─ not found → console.info "[mdx-ext] not logged in; library stripes disabled"
                      stripeRenderer becomes no-op (returns null status for every ID)
```

Token never written to `chrome.storage.local` (i.e. never persisted to disk via the extension). Lives only in `chrome.storage.session`. SW never reads localStorage directly; the token always comes from the content script.

If SW receives a 401 from `/manga/status`:
1. Purge `chrome.storage.session` token.
2. Send `{type: REQUEST_TOKEN_REFRESH}` to the requesting content script.
3. Content script re-runs `findJWT()` and re-sends `SET_TOKEN`.
4. SW retries the API call once. If still 401, gives up — stripeRenderer no-ops until next page load.

## Message protocol additions

Existing v0.1 messages stay. New:

| Type | Direction | Payload | Response |
|---|---|---|---|
| `SET_TOKEN` | content → SW | `{token: string}` | `{ok: true}` |
| `REQUEST_TOKEN_REFRESH` | SW → content | `{}` | (content sends `SET_TOKEN` async) |
| `GET_STATUSES` | content → SW | `{ids: string[]}` | `{statuses: {[id]: status \| null}}` |

`LOOKUP_MANGA` and `GET_STATUSES` are kept separate (different cache layers, different lifetimes — manga-attrs is 7 days local; library is 5 min session). Content script issues both in parallel after debouncing IDs.

## Cache strategy

Single key: `library:status` in `chrome.storage.session`:

```js
{
  statuses: { "<mangaId>": "reading" | "on_hold" | "plan_to_read" |
                            "completed" | "re_reading" | "dropped" },
  fetchedAt: <epoch ms>
}
```

TTL: **5 minutes**. The endpoint returns the entire library in one response, so refresh is a single fetch — cheap, no need to be conservative.

`chrome.storage.session` (not `local`):
- Auto-cleared on browser close — no stale library data persisted.
- Library is mildly sensitive (reveals reading habits); session-scoped storage keeps it ephemeral.

## Color scheme

| Status | Color | Hex |
|---|---|---|
| `reading` | green | `#16a34a` |
| `re_reading` | cyan | `#06b6d4` |
| `plan_to_read` | amber | `#f59e0b` |
| `completed` | indigo | `#6366f1` |
| `on_hold` | slate | `#64748b` |
| `dropped` | red | `#ef4444` |
| (none) | — | no stripe |

Implementation: a `::before` pseudo-element on the row anchor, `position: absolute; left: 0; top: 0; bottom: 0; width: 4px; background-color: <color>`. Anchor must be `position: relative` (we'll add it via the `mdx-ext-status` class). No DOM nodes added — just one className change per row.

## Scope

`scopeDetector` regex broadens to:

```js
const SCOPE = /^(\/(titles\/(latest|recent)|search|list\/[^/]+)\/?(\?.*)?|\/)$/;
```

Matches:
- `/titles/latest`
- `/titles/recent`
- `/search`
- `/list/{uuid}` (custom user lists)
- `/` (home page)

Excludes `/titles/follows/feed` (every row would be in the library — visual noise).

The row-detection logic from v0.1 is unchanged — still finds anchors via `a[href^="/title/"]` with non-empty text. The stripe goes on that same anchor element.

## Error handling

| Scenario | Behavior |
|---|---|
| No JWT in localStorage | `console.info` once, no stripes, no errors |
| JWT found but malformed | Same as no JWT |
| `/manga/status` returns 401 | Token-refresh flow (see Auth) |
| `/manga/status` returns 5xx / network error | Retain previous cache (if any); skip stripes for any uncached IDs |
| Token expired mid-session | One refresh attempt; if that fails, no stripes until next page reload |
| User logs out of MangaDex | Their JWT disappears from localStorage; next discovery returns null; stripes vanish on next page navigation |

## Files changed

- `manifest.json` — add `"storage"` already there; nothing new (`chrome.storage.session` is part of the existing storage permission).
- `src/content.js` — add `findJWT()`, `tokenSync`, `stripeRenderer`; broaden `SCOPE_PATTERN`.
- `src/background.js` — add `libraryFetcher`, `libraryCache`, `SET_TOKEN`/`GET_STATUSES` handlers, 401 refresh flow.
- `src/styles.css` — add `.mdx-ext-status` (relative positioning) + 6 `--{status}` color rules.

## Test plan

1. **Logged out** — load `/titles/latest`, verify no stripes appear and console shows the "not logged in" info message exactly once.
2. **Logged in, all 6 statuses present** — set one manga in each status via MangaDex's UI, then load `/titles/latest`. Verify each appears with its stripe color.
3. **Add a manga to a list mid-session** — set a manga to "Reading", wait 5 min, reload the page. Stripe should appear (cache expiry).
4. **Token expiry** — manually expire/edit the JWT in localStorage, reload page. Should see one info log; stripes for previously-cached IDs may still render until cache expires; new lookups gracefully no-op.
5. **Cross-page** — visit each scoped page and confirm stripes render. Visit `/titles/follows/feed` and confirm stripes do NOT render.
6. **No regression** — pills from v0.1+v0.2 still work alongside stripes.

## Open questions / risks

- **JWT discovery heuristic fragility**: if MangaDex stops storing JWTs in localStorage (e.g. switches to in-memory tokens or HttpOnly cookies), `findJWT()` returns null and stripes silently disable. We accept this; v0.3.1 fallback is the fetch-interception approach.
- **CORS**: SW already has `host_permissions` for `api.mangadex.org/*`; the same permission covers the auth-bearing endpoint. No new permission needed.
- **`/manga/status` endpoint shape**: assumed based on MangaDex API conventions. Will verify by hitting the endpoint with a real token early in implementation; if shape differs, this spec gets a small amendment.
